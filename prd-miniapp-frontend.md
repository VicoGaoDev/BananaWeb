# Banana 小程序前端 PRD

> 面向微信小程序的 `uni-app` 前端落地规划文档
> 目标：复用现有 Banana Web 后端能力，快速上线用户侧微信小程序
> 最后更新：2026-04-25

---

# 一、项目目标

## 1.1 项目背景

当前系统已具备完整的 Web 端能力：

- 创意模版浏览
- 文生图 / 图编辑
- 局部重绘
- 提示词反推
- 历史记录
- 积分体系
- 后台管理

现阶段计划新增一个独立的微信小程序前端，使用 `uni-app` 作为开发框架，尽量复用现有后端接口、任务体系和存储方案，降低重复开发成本。

本 PRD 不替代现有 `prd.md`，而是专门用于指导微信小程序前端的产品范围、技术落地和实施节奏。

## 1.2 项目目标

本项目目标如下：

1. 新建独立的 `uni-app` 小程序前端项目，不影响当前 Web 前端。
2. 复用现有 FastAPI 后端的大部分用户侧接口。
3. 小程序登录采用微信授权登录，并最终换取当前业务系统自己的 JWT。
4. 第一阶段仅上线用户侧核心能力，不做后台管理能力迁移。
5. 在保证可上线的前提下，优先复用现有 COS、任务轮询、积分扣费和历史记录机制。

## 1.3 成功标准

若满足以下条件，则视为第一期成功上线：

- 用户可在微信小程序中完成授权登录
- 用户可浏览模版并带入生成
- 用户可上传参考图并提交生成任务
- 用户可查看生成进度与结果
- 用户可查看历史记录
- 用户可看到个人积分和基础账户信息
- 后端无需重构现有任务系统与积分系统

# 二、产品定位与范围

## 2.1 产品定位

微信小程序定位为：

**面向普通用户的轻量化 AI 绘图入口**

它的核心职责不是替代现有 Web 管理后台，而是提供：

- 更低门槛的移动端访问入口
- 更顺滑的微信内传播和使用体验
- 更适合用户日常生成、查看结果和复用模版的场景

## 2.2 本期范围

本期纳入范围的功能：

1. 微信授权登录
2. 模版列表与模版详情
3. 文生图 / 图编辑
4. 历史记录
5. 用户中心
6. 积分展示
7. 小程序端图片上传
8. 结果图预览与保存

## 2.3 暂不纳入范围

以下能力不在第一期范围内：

1. 后台管理能力迁移
2. 小程序内用户注册 / 密码登录
3. 小程序内修改密码
4. 小程序内头像上传
5. 提示词反推
6. 局部重绘
7. 多角色后台入口
8. 模版管理、用户管理、接口管理、COS 配置

说明：

- 第一阶段目标是先跑通最核心的“登录 -> 上传 -> 生成 -> 历史 -> 保存图片”主路径。
- 提示词反推和局部重绘虽然在现有后端已存在，但在小程序端交互复杂度更高，建议在第二期接入。

# 三、技术策略

## 3.1 前端技术栈

小程序前端采用：

- `uni-app`
- Vue 3
- TypeScript
- Pinia
- 微信小程序原生能力

## 3.2 项目结构策略

在仓库根目录下新增独立目录：

```text
BananaWeb/
├── frontend/                 # 现有 Web 前端
├── backend/                  # 现有 FastAPI 后端
├── prd.md                    # 现有系统 PRD
└── prd-miniapp-frontend.md   # 小程序前端独立 PRD
```

后续建议新增：

```text
BananaWeb/
└── uniapp/                   # 微信小程序前端项目
```

## 3.3 后端复用策略

原则上复用现有用户侧后端接口，包括但不限于：

- `/api/auth/me`
- `/api/tasks`
- `/api/tasks/:id`
- `/api/history`
- `/api/templates`
- `/api/templates/:id`
- `/api/upload/credential`
- `/api/images/:id/download`
- `/api/images/:id/regenerate`

其中最主要的新增项是：

**补充微信小程序登录桥接接口**

推荐新增：

- `POST /api/auth/wechat/login`

该接口负责：

1. 接收小程序 `code`
2. 调用微信登录凭证校验接口
3. 获取 `openid`
4. 查找或创建本地业务用户
5. 签发当前系统自己的 JWT

这样做的好处是：

- 小程序只在登录时使用微信身份
- 登录完成后仍统一走现有 JWT 认证体系
- 后续所有业务接口可与 Web 端保持一致

## 3.4 登录策略

本期登录方案采用：

**微信授权登录 + 后端签发业务 Token**

不采用以下方案作为一期主方案：

- 直接复用 Web 端账号密码登录
- 直接以 CloudBase Web 邮箱注册体系作为主认证方式
- 让前端仅凭微信侧身份直接访问业务接口

原因：

1. 当前业务系统权限和用户数据均在自有后端数据库中
2. 后端现有接口已经基于 JWT 鉴权
3. 小程序端若直接与微信身份强耦合，会增加后续权限体系维护成本

## 3.5 存储与上传策略

小程序端继续采用现有 COS 直传思路：

1. 小程序向后端请求上传凭证
2. 后端返回腾讯云 COS 临时凭证
3. 小程序直传 COS
4. 小程序将最终图片 URL 提交给任务接口

说明：

- 第一版若直传适配成本过高，可先使用后端文件上传接口做开发期兜底
- 正式上线建议仍以 COS 直传为主，降低后端带宽与存储压力

# 四、用户角色与使用场景

## 4.1 目标用户

本期仅面向普通用户，不面向管理员。

典型用户包括：

- 需要快速生成图片的个人用户
- 在微信群、朋友圈传播场景中使用 AI 绘图能力的用户
- 希望在手机端直接查看和保存生成结果的用户

## 4.2 典型使用场景

### 场景一：模版带入生成

1. 用户进入小程序
2. 浏览创意模版
3. 打开模版详情
4. 点击“使用此模版”
5. 自动带入生成页
6. 提交生成任务
7. 查看结果并保存图片

### 场景二：直接自定义生成

1. 用户进入生成页
2. 输入提示词
3. 上传参考图
4. 选择尺寸和数量
5. 提交任务
6. 轮询等待结果
7. 预览并保存图片

### 场景三：查看历史记录

1. 用户进入历史页
2. 查看近期任务
3. 打开历史详情
4. 预览大图
5. 保存图片或重新生成

# 五、功能需求

## 5.1 登录与账户

### 目标

让用户以最低成本完成登录，进入业务体系。

### 功能要求

1. 小程序首次进入时可匿名浏览模版
2. 当用户执行以下动作时要求登录：
   - 开始生成
   - 上传参考图
   - 查看历史记录
   - 保存结果图
3. 登录入口采用微信授权登录
4. 登录成功后保存业务 JWT 与用户信息
5. 若 Token 失效，自动清空登录态并重新拉起登录

### 页面形态

- 不单独设计复杂登录页
- 使用轻量授权页或弹层引导用户完成微信登录

### 本期不做

- 密码登录
- 邮箱注册
- 修改密码
- 头像上传

## 5.2 模版模块

### 目标

让用户以更低决策成本进入生成流程。

### 功能要求

1. 默认首页为模版列表
2. 支持标签筛选
3. 支持模版卡片列表展示
4. 点击卡片进入模版详情
5. 模版详情展示：
   - 结果图
   - 提示词
   - 参考图
   - 宽高比
   - 分辨率
6. 支持“使用此模版”
7. 点击后将参数带入生成页，但不自动提交

### 说明

模版接口优先复用现有接口，避免单独为小程序设计新数据结构。

## 5.3 自定义绘图

### 目标

在小程序内提供最核心的 AI 生图能力。

### 本期保留能力

1. 提示词输入
2. 模型选择
3. 参考图上传
4. 图片数量选择
5. 宽高比选择
6. 分辨率选择
7. 提交生成任务
8. 展示扣费提示
9. 轮询获取结果

### 本期不做能力

1. 提示词反推
2. 局部重绘
3. 复杂图片编辑交互

### 交互要求

1. 生成前校验登录态
2. 生成前校验积分是否足够
3. 提交后展示任务中状态
4. 若任务成功，展示结果列表
5. 若任务失败，展示明确失败提示

## 5.4 历史记录

### 目标

让用户可以持续复用过去的成果与提示词。

### 功能要求

1. 支持查看历史记录列表
2. 支持按时间倒序展示
3. 历史卡片展示：
   - 缩略图
   - 状态
   - 时间
   - 简短提示词
4. 支持查看详情
5. 支持预览大图
6. 支持保存图片
7. 支持重新生成

### 本期可延后能力

- 复杂筛选
- 删除历史记录
- 重新编辑整套参数回填

## 5.5 用户中心

### 目标

提供基础账户信息与状态管理能力。

### 功能要求

1. 展示头像、昵称或用户名
2. 展示当前积分
3. 展示登录状态
4. 支持退出登录
5. 支持跳转历史记录

### 本期不做

- 修改密码
- 完整账户设置页
- 账号绑定与解绑

## 5.6 图片保存

### 目标

让用户能在微信内保存生成结果。

### 功能要求

1. 结果图支持大图预览
2. 支持保存到系统相册
3. 首次保存时处理相册权限申请
4. 保存失败时给出明确提示

# 七、接口规划

## 7.1 可直接复用的接口

可优先复用以下现有接口：

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/api/templates` | 模版列表 |
| GET | `/api/templates/:id` | 模版详情 |
| GET | `/api/config/task-scenes` | 获取积分和场景配置 |
| POST | `/api/tasks` | 创建生成任务 |
| GET | `/api/tasks/:id` | 查询任务结果 |
| GET | `/api/history` | 获取历史记录 |
| POST | `/api/upload/credential` | 获取 COS 上传凭证 |
| GET | `/api/auth/me` | 获取当前用户信息 |
| POST | `/api/images/:id/regenerate` | 单张重新生成 |

## 7.2 需要新增的接口

本期建议新增：

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/api/auth/wechat/login` | 微信小程序登录，返回业务 JWT |

建议请求示例：

```json
{
  "code": "wx_login_code"
}
```

建议响应示例：

```json
{
  "token": "jwt_token",
  "user": {
    "id": 1,
    "username": "wx_user_xxx",
    "role": "user",
    "avatar_url": "",
    "credits": 100
  }
}
```

## 7.3 数据兼容要求

小程序端应兼容以下图片 URL 形式：

1. COS 绝对 URL
2. 后端返回的相对路径 URL

小程序端需要统一封装图片地址解析逻辑，避免页面层重复判断。

第一期必需改造项如下：

1. 新增微信小程序登录接口
2. 用户表支持微信身份映射
3. 登录成功后返回现有业务 Token

## 8.2 用户身份映射建议

建议优先采用快速方案：

在 `users` 表增加字段：

- `wechat_openid`
- `wechat_unionid`

说明：

- 若后续存在多身份体系扩展需求，再考虑拆分独立身份表
- 第一版优先降低实现复杂度

## 8.3 非必需改造

以下可在后续迭代中决定是否补充：

- 手机号绑定接口
- 微信用户资料更新接口
- 小程序专用统计接口

# 九、小程序交互与体验要求

## 9.1 交互原则

1. 路径尽量短
2. 页面层级尽量浅
3. 结果反馈尽量即时
4. 异常状态可理解

## 9.2 关键体验要求

### 登录体验

- 未登录用户允许浏览模版
- 需要登录时再拉起授权
- 避免一进入小程序就强制弹授权

### 生成体验

- 提交后立即展示“生成中”状态
- 支持轮询刷新
- 结果返回后自动更新列表

### 保存体验

- 用户保存图片时给出明显反馈
- 若权限未开启，明确引导用户去设置页开启

### 异常体验

以下情况需要明确文案：

- 登录失败
- 上传失败
- 任务创建失败
- 任务轮询超时
- 图片保存失败
- 网络异常

# 十、非功能需求

## 10.1 性能要求

1. 首屏应优先展示模版列表骨架屏或加载态
2. 列表图片建议使用缩略图
3. 历史记录与模版列表需要分页或懒加载

## 10.2 安全要求

1. 小程序端不得保存敏感密钥
2. COS 密钥仍由后端 STS 临时签发
3. 用户态接口继续使用 JWT

## 10.3 兼容要求

1. 目标平台优先支持微信小程序
2. 第一版不要求同时适配 H5、App 和其他小程序平台

# 十二、风险与依赖

## 12.1 主要风险

1. 微信登录桥接接口未及时完成，导致前端无法打通登录闭环
2. 小程序端 COS 上传适配复杂度高于预期
3. 合法域名、HTTPS、图片域名配置未提前完成
4. 小程序图片保存权限处理不当，影响用户体验

## 12.2 外部依赖

1. 微信小程序 `AppID`
2. 微信小程序后台合法域名配置
3. CloudBase 环境
4. 腾讯云 COS 正式可用配置
5. 后端服务具备公网 HTTPS 访问能力
